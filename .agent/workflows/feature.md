# Feature Development Workflow

이 워크플로우는 사용자의 요구사항을 명확히 하고, 체계적인 계획 수립 후 작업을 시작하기 위해 설계되었습니다.

## 1. 요구사항 분석 및 협의 (Analysis & Discussion)

1. 사용자가 `/feature [기능명]`으로 요청하면, 해당 기능에 대한 **구체적인 요구사항**을 인터뷰합니다.
   - **질문 예시**:
     - "해당 기능은 구체적으로 어떤 페이지나 컴포넌트에 추가되나요?"
     - "기대하는 사용자 인터랙션이나 시각적 디자인이 있나요?"
     - "기술적인 제약사항이나 특정 라이브러리 사용 선호가 있나요?"
2. 사용자의 답변을 바탕으로 **작업 개요**를 정리하고, AI 개발자로서 **개선 사항이나 놓친 부분**을 제안합니다.
   - **제안 예시**: "접근성을 위해 ARIA 속성도 같이 추가하는 것이 좋겠습니다.", "모바일 반응형 동작은 어떻게 처리할까요?"
3. 최종적으로 합의된 작업 범위를 요약하여 사용자에게 확인을 받습니다.

## 2. 구현 계획 수립 (Planning)

1. 작업 범위가 확정되면 `implementation_plan.md` 아티팩트를 작성하여 상세 계획을 수립합니다.
   - **필수 포함 내용**:
     - **개요 (Goal)**: 작업의 목적과 배경을 언제 누가 봐도 이해할 수 있게 간략히 작성
     - **작업 단계 (Phases)**: 작업이 복잡할 경우 Phase를 나누어 단계별로 작성 (예: UI 구현 -> 로직 연동 -> 테스트)
     - **변경 사항 (Changes)**: 수정/생성할 파일 목록 및 주요 변경점
   - **테스트 케이스 (Verification Plan)**:
     - 사용자가 작업을 완료한 후 직접 테스트해 볼 수 있도록 상세한 **테스트 케이스**를 기재합니다.
     - 예: "1. [페이지] 접속 -> [버튼] 클릭 -> [결과] 확인"
2. `notify_user` 툴을 사용하여 사용자에게 계획을 검토(Review) 요청합니다.
   - 한글로 작성한다

## 3. 브랜치 생성 및 작업 시작 (Execution)

1. 사용자가 계획을 승인하면 작업에 적합한 **기능 브랜치**를 생성합니다.
   - **네이밍 규칙**: `feature/[기능-이름]` (kebab-case 사용)
   - **명령어**:
     ```bash
     git checkout -b feature/my-feature-name
     ```
2. `task_boundary` 툴을 사용하여 작업을 시작했음을 시스템에 알립니다.
3. 계획에 따라 코딩 작업을 진행합니다.

## 4. 완료 및 인수인계 (Completion & Handover)

1. 모든 구현이 완료되면 commit, push, merge는 진행하지 않습니다.
2. 대신, 사용자가 직접 기능을 검증할 수 있도록 **상세한 테스트 가이드(test_case.md)** 파일을 생성합니다.
   - **파일 경로**: 해당 기능 디렉토리 또는 루트의 `test_case.md` (혹은 적절한 위치)
   - **내용**:
     - 기능 설명
     - 테스트 준비 사항 (환경 설정 등)
     - 단계별 테스트 시나리오
     - 예상 결과 스크린샷 등이 있다면 포함
3. 사용자에게 작업 완료를 알리고, `test_case.md`를 참고하여 테스트를 진행해 달라고 요청합니다.
